Why can't we use arrays instead of MongoDB?
A strong answer is:
Arrays are stored in RAM and lose data when the application restarts. They are also not designed for persistent storage, efficient querying of large datasets, concurrent access by multiple servers, indexing, or scalability. Databases solve these problems by storing data permanently and providing optimized mechanisms to manage and retrieve it.

A collection is simply: A group of similar documents.
Inside every document...Each key is called a field like name,age,salary type of a single object , hink of fields as the properties of a JavaScript object.
A collection is a group of related documents, similar to a table in SQL. A document is an individual record stored inside a collection and is represented as a JSON-like object.
A collection behaves somewhat like an array of documents conceptually, but it lives inside MongoDB, supports indexing, querying, persistence, and many database features. It's much more powerful than a JavaScript array.

Filtering or searching is same thing 
READ is the operation of retrieving existing documents from the database. It can return all documents or only those matching specific query conditions.
UPDATE modifies an existing document in the database without creating a new document. MongoDB first identifies the target document and then updates the specified fields.
DELETE removes an existing document from the database. MongoDB first identifies the target document and then permanently removes it from the collection.

Why is Server Started printed before MongoDB Connected Successfully?
Because mongoose.connect() is asynchronous. It sends the connection request and returns immediately. JavaScript continues executing the next line (app.listen()), so the server starts first. Once the database connection is established, the Promise resolves and the .then() callback runs.
